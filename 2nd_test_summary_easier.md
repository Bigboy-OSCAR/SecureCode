## 1. 가장 쉬운 한 줄 결론

이번 실험들의 결론은 아주 단순하다.

```text
큰 자료를 LLM에게 통째로 주지 말자.
필요한 부분만 찾아서 작게 주자.
```

큰 책 한 권에서 한 문장만 필요할 때, 책 전체를 통째로 외우게 할 필요는 없다.

차라리 이렇게 하는 것이 좋다.

```text
큰 책은 책장에 둔다.
책 이름과 쪽 번호만 알려준다.
필요한 쪽만 펼쳐서 읽는다.
```

이번 실험의 핵심이 바로 이것이다.

## 2. 먼저 알아야 할 비유

LLM은 질문에 답하는 똑똑한 친구라고 생각하면 된다.

하지만 이 친구에게도 한계가 있다. 책상 위에 올릴 수 있는 종이 양이 정해져 있다.

### 2.1 LLM의 책상

LLM이 한 번에 읽을 수 있는 공간을 `context window`라고 부른다.

쉬운 말로 하면 다음과 같다.

```text
context window = LLM의 책상 크기
```

책상이 작으면 너무 많은 종이를 한 번에 올릴 수 없다.

### 2.2 큰 자료를 통째로 올리면 생기는 문제

코드 어시스턴트는 이런 자료를 읽어야 한다.

- 긴 로그
- 큰 JSON 파일
- 많은 코드 파일
- 테스트 실패 결과
- 프로젝트 문서
- 실행 중 만들어진 큰 결과물

이 자료를 전부 LLM의 책상 위에 올리면 문제가 생긴다.

1. 책상보다 커서 못 올린다.
2. 올리더라도 읽는 데 오래 걸린다.
3. 중요한 부분을 못 찾을 수 있다.
4. 비슷하지만 틀린 값을 보고 헷갈릴 수 있다.

### 2.3 Pointer는 번호표다

`pointer`는 큰 물건에 붙이는 번호표라고 생각하면 된다.

큰 상자를 LLM에게 직접 주지 않는다.

대신 이렇게 한다.

```text
상자에 번호표를 붙인다.
LLM에게는 번호표만 보여준다.
필요할 때 도구가 번호표를 보고 상자를 찾아간다.
상자 안에서 필요한 물건만 꺼낸다.
```

이번 실험에서는 세 종류의 pointer가 나온다.

| pointer 종류 | 쉬운 설명 | 예시 |
|---|---|---|
| source pointer | 이미 파일로 있는 큰 자료의 번호표 | `source://log/prod.log` |
| runtime pointer | 실행 중 새로 생긴 큰 결과물의 번호표 | `runtime://log/run_tests/0001` |
| code pointer | 코드 안의 특정 함수나 클래스 번호표 | `memory://symbol/app.py::login` |

## 3. 네 실험을 한눈에 보기

| 실험 | 쉬운 비유 | 확인한 것 | 결과 |
|---|---|---|---|
| Runtime Pointer | 방금 나온 큰 출력물에 번호표 붙이기 | 실행 중 생긴 큰 결과를 통째로 넣지 않아도 되는가? | 된다. pointer 방식은 3/3 성공했다. |
| Self-Extend | LLM의 책상을 더 크게 만들기 | 책상만 키우면 정답을 잘 찾는가? | 책상은 커졌지만 정답은 잘 못 찾았다. |
| Source Pointer | 이미 있는 큰 파일에 번호표 붙이기 | 큰 파일을 통째로 넣지 않아도 되는가? | 된다. pointer 방식은 3/3 성공했다. |
| Symbol Bundle | 코드 전체 대신 필요한 코드 묶음만 주기 | 코드에서 필요한 함수와 주변 정보만 주면 되는가? | 찾기는 좋아졌지만, 계산 실수 때문에 최종 정답은 1/3이었다. |

가장 중요한 차이는 이것이다.

| 방식 | 쉬운 설명 | 실험에서 본 모습 |
|---|---|---|
| 전체를 통째로 넣기 | 큰 책 전체를 책상 위에 올리기 | 자주 실패하거나 오래 걸렸다. |
| pointer 쓰기 | 책은 책장에 두고 쪽 번호만 쓰기 | 작고 빠르게 필요한 부분을 찾았다. |
| Self-Extend | 책상을 크게 만들기 | 더 많이 올릴 수는 있었지만, 답을 잘 찾지는 못했다. |
| Symbol Bundle | 필요한 코드 조각만 묶어 주기 | 읽을 양은 줄었지만, 모델이 계산을 틀릴 수 있었다. |

## 4. Runtime Pointer 실험

### 4.1 이 실험은 무엇인가

Runtime Pointer 실험은 "실행 중 생긴 큰 결과물"을 다룬다.

예를 들어 테스트를 실행하면 긴 로그가 나온다.

```text
테스트 실행
-> 엄청 긴 로그 생성
```

이 로그를 LLM에게 통째로 주면 너무 크다.

그래서 실험에서는 이렇게 했다.

```text
긴 로그를 RuntimeStore라는 저장소에 넣는다.
로그에 runtime://... 번호표를 붙인다.
LLM에게는 번호표와 짧은 설명만 준다.
필요하면 도구가 번호표를 열고 필요한 줄만 꺼낸다.
```

### 4.2 왜 필요한가

코드 어시스턴트가 실제로 일할 때는 큰 실행 결과가 자주 생긴다.

- 테스트 실패 로그
- 빌드 로그
- API 응답
- 정적 분석 결과
- 실행 중 만든 JSON

이것을 모두 LLM에게 넣으면 책상이 꽉 찬다.

그래서 "큰 결과물은 따로 보관하고, 필요한 부분만 꺼내자"는 생각을 확인했다.

### 4.3 어떻게 실험했는가

세 가지 방식을 비교했다.

| 방식 | 쉬운 설명 |
|---|---|
| `full_runtime` | 큰 실행 결과를 LLM에게 통째로 준다. |
| `runtime_pointer_oracle` | 사람이 이미 정답 위치를 안다고 치고, 필요한 부분만 꺼내서 준다. |
| `runtime_pointer_select` | LLM이 어떤 도구를 쓸지 고르고, 그 도구가 필요한 부분만 꺼내 준다. |

실험에 사용한 큰 실행 결과는 세 가지였다.

| 자료 | 쉬운 설명 | 질문 |
|---|---|---|
| checkout trace log | 결제 흐름 로그 | 특정 요청이 왜 실패했는지 찾기 |
| operations runbook | 운영 설명 문서 | 특정 설정값 찾기 |
| tenant snapshot dict | 큰 설정 데이터 | 특정 고객 설정값 찾기 |

자료 크기는 작게 만든 버전과 크게 만든 버전이 있었다.

| 자료 | small 크기 | medium 크기 |
|---|---:|---:|
| checkout trace log | 79,071 글자 | 608,116 글자 |
| operations runbook | 9,007 글자 | 59,599 글자 |
| tenant snapshot dict | 134,023 글자 | 891,159 글자 |

필요한 부분을 꺼내기 위해 쓴 도구는 다음과 같다.

| 도구 | 하는 일 |
|---|---|
| `grep_runtime_log` | 긴 로그에서 필요한 줄 주변만 찾는다. |
| `extract_runtime_doc_section` | 긴 문서에서 필요한 section만 찾는다. |
| `extract_runtime_json_path` | 큰 JSON/dict에서 필요한 값만 찾는다. |

### 4.4 결과는 어땠는가

Small 결과:

| 방식 | 성공률 | 평균 시간 | 평균 prompt token |
|---|---:|---:|---:|
| `full_runtime` | 1/3 | 4.961초 | 18,785 |
| `runtime_pointer_oracle` | 3/3 | 2.768초 | 311 |
| `runtime_pointer_select` | 3/3 | 7.050초 | 756 |

Medium 결과:

| 방식 | 성공률 | 평균 시간 | 평균 prompt token |
|---|---:|---:|---:|
| `full_runtime` | 1/3 | 26.270초 | 131,136 |
| `runtime_pointer_oracle` | 3/3 | 2.610초 | 311 |
| `runtime_pointer_select` | 3/3 | 6.891초 | 757 |

여기서 중요한 점은 이것이다.

```text
자료가 커져도 pointer 방식의 prompt token은 거의 늘지 않았다.
```

| 방식 | small prompt token | medium prompt token |
|---|---:|---:|
| `runtime_pointer_oracle` | 311 | 311 |
| `runtime_pointer_select` | 756 | 757 |

반대로 `full_runtime`은 큰 자료를 통째로 넣었기 때문에 log와 JSON에서 context overflow가 났다.

### 4.5 아주 쉽게 해석하면

`full_runtime`은 큰 책을 LLM 책상 위에 통째로 올린 것이다.

그래서 책상이 넘쳤다.

`runtime_pointer`는 책은 책장에 두고, 필요한 쪽만 펼쳐서 보여준 것이다.

그래서 작고 빠르게 답을 찾았다.

### 4.6 한계는 무엇인가

좋은 결과가 나왔지만 아직 조심할 점이 있다.

- 실험용으로 만든 가짜 데이터였다.
- 실제 로그나 API 결과는 더 지저분할 수 있다.
- LLM이 고른 도구 사용법이 틀릴 수 있다.
- 실행 중 저장한 큰 결과물을 언제 지울지 정해야 한다.
- 민감한 정보가 들어 있으면 권한 검사와 마스킹이 필요하다.

### 4.7 코드 어시스턴트에는 어떻게 쓰는가

실제 코드 어시스턴트에서는 이렇게 쓸 수 있다.

```text
테스트 실행
-> 긴 실패 로그가 나옴
-> 로그 전체를 LLM에게 주지 않음
-> runtime://... 번호표로 저장
-> 실패 줄, 에러 줄, stack trace만 꺼냄
-> 그 작은 증거로 관련 코드를 찾음
-> 수정안을 만든다
```

## 5. Self-Extend 2차 실험

### 5.1 이 실험은 무엇인가

Self-Extend는 LLM의 책상을 더 크게 만드는 방법이라고 생각하면 된다.

보통 LLM 책상에는 종이를 4장쯤 올릴 수 있다고 하자.

Self-Extend는 이렇게 말한다.

```text
책상을 2배, 4배로 넓혀 보자.
그러면 긴 글도 읽을 수 있지 않을까?
```

이번 실험은 여기서 한 걸음 더 나아갔다.

```text
책상이 커지면,
긴 글 속에 숨어 있는 정답도 잘 찾을까?
```

이 질문을 확인했다.

### 5.2 어떻게 실험했는가

세 가지 테스트를 했다.

| 테스트 | 쉬운 설명 |
|---|---|
| Passkey Retrieval | 긴 글 속에 숫자 `16808`을 숨기고, 나중에 그 숫자를 물어본다. |
| PG-19 PPL | 긴 문학 글을 모델이 얼마나 자연스럽게 예측하는지 본다. |
| 코드 어시스턴스 benchmark | 긴 코드 더미 속에서 함수 결과, 호출 함수, 테스트 실패 원인을 찾게 한다. |

Passkey 테스트는 이렇게 생겼다.

```text
긴 의미 없는 글
중간 어딘가에 16808 숨김
마지막에 질문: 아까 숨긴 숫자가 뭐야?
```

코드 benchmark는 세 문제였다.

| 문제 | 물어본 것 | 정답 |
|---|---|---|
| 함수 결과 찾기 | `derive_rotation_marker("apac", 7)` 결과 | `ROTATE::APAC::0007` |
| 호출 함수 찾기 | `derive_rotation_marker`를 부르는 함수 | `schedule_rotation` |
| 테스트 실패 원인 찾기 | 왜 zero padding이 사라지는가 | `.lstrip("0")` 때문 |

### 5.3 Passkey 결과

| 모델 | 조건 | 성공 | 쉬운 해석 |
|---|---:|---:|---|
| Q4_K_M | off | 2/3 | 4k 근처는 맞혔지만, 4k 초과는 넘쳤다. |
| Q4_K_M | G=2 | 0/6 | 더 긴 입력은 받았지만 숫자를 못 찾았다. |
| Q4_K_M | G=4 | 0/2 | 더 긴 입력은 받았지만 출력이 무너졌다. |
| Q5_K_M | off | 2/3 | Q4와 비슷했다. |
| Q5_K_M | G=2 | 0/6 | Q4와 비슷하게 실패했다. |
| Q5_K_M | G=4 | 0/2 | 출력이 무너졌다. |

Self-Extend를 켜면 책상 크기는 커졌다.

| 조건 | 책상 크기처럼 볼 수 있는 `n_ctx` |
|---|---:|
| off | 4320 |
| G=2 | 8416 |
| G=4 | 16608 |

하지만 중요한 숫자 `16808`을 찾지는 못했다.

즉 다음과 같다.

```text
책상은 커졌다.
하지만 숨은 물건을 잘 찾지는 못했다.
```

### 5.4 PG-19 PPL 결과

PG-19는 긴 글을 얼마나 자연스럽게 예측하는지 보는 실험이다.

다만 이번에는 도구가 Self-Extend 옵션을 지원하지 않았다. 그래서 Self-Extend on/off 비교가 아니라 Q4와 Q5의 기본 점수만 봤다.

| 모델 | 조건 | PPL | 쉬운 해석 |
|---|---:|---:|---|
| Q4_K_M | 2048 | 7.8529 | 정상 범위 |
| Q5_K_M | 2048 | 7.8592 | Q4와 거의 같음 |
| Q4_K_M | 4096 | 304.6347 | 학습 길이를 넘겨서 점수가 크게 나빠짐 |
| Q5_K_M | 4096 | 309.3053 | Q4와 비슷하게 나빠짐 |

여기서 조심해야 할 점은 이것이다.

```text
이 PG-19 결과는 Self-Extend 성능 비교가 아니다.
도구가 Self-Extend 옵션을 지원하지 않았기 때문이다.
```

| 모델 | 조건 | 성공 | 쉬운 해석 |
|---|---:|---:|---|
| Q4_K_M | off | 0/3 | 입력이 너무 길어서 못 받았다. |
| Q4_K_M | G=2 | 0/3 | 입력은 받았지만 답을 못 찾았다. |
| Q4_K_M | G=4 | 0/3 | 빈 출력이 나왔다. |
| Q5_K_M | off | 0/3 | 입력이 너무 길어서 못 받았다. |
| Q5_K_M | G=2 | 0/3 | 입력은 받았지만 답을 못 찾았다. |
| Q5_K_M | G=4 | 0/3 | 빈 출력이 나왔다. |

코드 어시스턴트 입장에서 중요한 결론은 이렇다.

```text
긴 코드를 넣을 수 있게 되어도,
그 안에서 필요한 함수나 원인을 정확히 찾는 것은 다른 문제다.
```

### 5.6 아주 쉽게 해석하면

Self-Extend는 큰 책상을 만드는 실험이다.

큰 책상은 도움이 될 수 있다.

하지만 이번 결과에서는 책상이 커졌다고 해서 물건을 잘 찾지는 못했다.

코드 어시스턴트에게는 이것이 더 중요하다.

```text
책상을 크게 만드는 것보다
필요한 종이만 골라서 올리는 것이 더 중요하다.
```

### 5.7 한계는 무엇인가

- 사용한 도구가 논문 구현과 완전히 같다고 보장할 수 없다.
- PG-19에서는 Self-Extend 옵션을 켜고 비교하지 못했다.
- Q8_0 같은 더 높은 정밀도 모델은 테스트하지 않았다.
- 반복 횟수가 많지 않았다.
- 코드 benchmark는 실제 큰 프로젝트가 아니라 실험용으로 만든 repo였다.
- 출력이 왜 무너졌는지 원인을 완전히 분리하지 못했다.

### 5.8 코드 어시스턴트에는 어떻게 쓰는가

현재 결과만 보면 Self-Extend를 기본 기능으로 쓰기는 어렵다.

대신 이렇게 쓰는 것이 현실적이다.

```text
먼저 rg, symbol index, 로그 검색으로 필요한 부분을 찾는다.
작게 줄인 context를 LLM에게 준다.
그래도 너무 길면 Self-Extend를 실험적으로 써 본다.
답을 만들기 전에 함수명, 파일명, 테스트명, 원인 줄을 다시 확인한다.
틀리면 더 작게 줄여서 다시 시도한다.
```

## 6. Source Pointer 실험

### 6.1 이 실험은 무엇인가

Source Pointer 실험은 "이미 파일로 존재하는 큰 자료"를 다룬다.

예를 들어 disk에 큰 로그 파일이 있다고 하자.

```text
checkout_gateway.log
```

이 파일 전체를 LLM에게 주지 않는다.

대신 파일에 번호표를 붙인다.

```text
source://log/logs/checkout_gateway.log
```

LLM은 번호표를 보고 필요한 도구를 고른다. 도구는 그 파일에서 필요한 부분만 꺼낸다.

### 6.2 Runtime Pointer와 무엇이 다른가

둘 다 번호표를 쓰지만 가리키는 대상이 다르다.

| 종류 | 가리키는 것 | 쉬운 비유 |
|---|---|---|
| source pointer | 이미 disk에 있는 파일 | 책장에 원래 있던 책의 위치표 |
| runtime pointer | 실행 중 새로 생긴 결과 | 방금 인쇄한 종이에 붙인 번호표 |

### 6.3 어떻게 실험했는가

세 가지 방식을 비교했다.

| 방식 | 쉬운 설명 |
|---|---|
| `full_source` | 큰 파일 전체를 LLM에게 준다. |
| `pointer_oracle` | 사람이 이미 정답 위치를 안다고 치고, 필요한 부분만 꺼내서 준다. |
| `pointer_select` | LLM이 어떤 도구를 쓸지 고르고, 그 도구가 필요한 부분만 꺼내 준다. |

사용한 파일은 세 종류였다.

| 파일 종류 | small 크기 | medium 크기 | 질문 |
|---|---:|---:|---|
| checkout gateway log | 80K | 596K | 특정 요청의 실패 이유 찾기 |
| operations runbook markdown | 12K | 56K | 특정 설정값 찾기 |
| tenant snapshot JSON | 116K | 752K | 특정 고객 설정값 찾기 |

필요한 부분을 꺼내는 도구는 다음과 같았다.

| 도구 | 하는 일 |
|---|---|
| `grep_log` | 긴 로그에서 필요한 줄 주변만 찾는다. |
| `extract_doc_section` | 긴 문서에서 필요한 section만 찾는다. |
| `extract_json_field` | 큰 JSON에서 필요한 값만 찾는다. |

### 6.4 결과는 어땠는가

Small 결과:

| 방식 | 성공률 | 평균 시간 | 평균 prompt token |
|---|---:|---:|---:|
| `full_source` | 1/3 | 54.401초 | 18,369 |
| `pointer_oracle` | 3/3 | 2.472초 | 286 |
| `pointer_select` | 3/3 | 6.430초 | 778 |

Medium 결과:

| 방식 | 성공률 | 평균 시간 | 평균 prompt token |
|---|---:|---:|---:|
| `full_source` | 1/3 | 22.117초 | 120,208 |
| `pointer_oracle` | 3/3 | 2.443초 | 286 |
| `pointer_select` | 3/3 | 6.423초 | 780 |

pointer 방식은 파일이 커져도 prompt token이 거의 늘지 않았다.

| 방식 | small prompt token | medium prompt token |
|---|---:|---:|
| `pointer_oracle` | 286 | 286 |
| `pointer_select` | 778 | 780 |

반대로 `full_source`는 문제가 있었다.

- small log에서 context overflow가 났다.
- small JSON에서는 LLM이 정답이 아니라 비슷한 오답을 골랐다.
- medium log와 JSON에서는 context overflow가 났다.
- medium 문서 task는 성공했지만 16,770 prompt tokens와 63.68초가 필요했다.

같은 medium 문서 질문을 `pointer_oracle`로 처리하면 217 prompt tokens와 2.12초로 끝났다.

### 6.5 아주 쉽게 해석하면

`full_source`는 책 한 권을 통째로 책상 위에 올린 것이다.

`source pointer`는 책장에서 필요한 쪽만 복사해서 올린 것이다.

그래서 pointer 방식이 더 작고 빠르고 정확했다.

### 6.6 한계는 무엇인가

- 실험용으로 만든 가짜 파일이었다.
- 실제 파일은 더 지저분할 수 있다.
- PDF, CSV, HTML, DB 결과 같은 다른 형식도 더 다뤄야 한다.
- 파일 후보가 많아지면 LLM이 어떤 파일과 도구를 쓸지 고르기 어려울 수 있다.
- 외부 증거를 찾은 뒤에는 결국 관련 코드를 찾아야 한다.

### 6.7 코드 어시스턴트에는 어떻게 쓰는가

예를 들어 사용자가 이렇게 말한다고 하자.

```text
운영 로그에서 결제 실패 원인을 찾아서 고쳐줘.
```

좋은 흐름은 다음과 같다.

```text
로그 전체를 LLM에게 주지 않는다.
source://... 번호표로 로그를 가리킨다.
request_id 주변 줄만 꺼낸다.
그 줄에서 에러 원인을 본다.
symbol index로 관련 결제 함수를 찾는다.
필요한 코드 조각만 읽고 수정한다.
```

## 7. Symbol Bundle / Filtered Selector 2차 실험

### 7.1 이 실험은 무엇인가

Symbol 실험은 코드 전체를 다 읽지 않기 위한 실험이다.

코드에는 이름 붙은 조각이 많다.

- 함수
- 클래스
- 메서드
- 상수

이런 이름 붙은 코드 조각을 `symbol`이라고 부른다.

코드 어시스턴트가 질문에 답하려면 보통 코드 전체가 아니라 몇 개의 symbol만 보면 된다.

예를 들어 이런 질문이 있다고 하자.

```text
login 함수가 토큰 만료 때 어떤 에러를 내?
```

이때 프로젝트 전체를 다 읽을 필요는 없다.

`login` 함수와 그 함수가 쓰는 상수 정도만 보면 된다.

### 7.2 이전 방식의 문제

이전 방식은 대략 이랬다.

```text
전체 symbol 목록을 LLM에게 준다.
LLM이 그중 하나를 고른다.
선택한 symbol 하나만 읽는다.
답한다.
```

문제가 두 개 있었다.

1. 전체 symbol 목록이 너무 크다.
2. 함수 하나만 읽으면 주변 상수나 helper 함수를 놓칠 수 있다.

그래서 2차 실험에서는 이렇게 바꿨다.

```text
질문과 관련 있어 보이는 후보만 먼저 고른다.
LLM은 작은 후보 목록에서 하나를 고른다.
선택한 함수뿐 아니라 주변 상수와 helper 함수도 함께 묶는다.
그 묶음을 LLM에게 준다.
```

이 묶음을 `symbol bundle`이라고 부른다.

### 7.3 어떻게 실험했는가

어려운 문제 3개를 사용했다.

함수 하나만 보면 틀리기 쉬운 문제들이다.

| 문제 | 물어본 것 | 추가로 필요한 것 |
|---|---|---|
| expired token 에러 문자열 | `validate_token`이 내는 정확한 문자열 | `TOKEN_EXPIRED_MESSAGE` 상수 |
| cache key 계산 | `build_cache_key(...)`의 최종 결과 | `sanitize_report_name` helper |
| billing 상태와 수수료 | `invoice_status(4)`와 `compute_late_fee(...)` 결과 | 함수와 메서드 둘 다 필요 |

두 가지 방식을 비교했다.

| 방식 | 쉬운 설명 |
|---|---|
| `symbol_bundle_oracle` | 사람이 정답 symbol을 이미 안다고 치고, 바로 bundle을 만든다. |
| `symbol_filtered_select_bundle` | 후보를 먼저 줄이고, LLM이 symbol을 고른 뒤 bundle을 만든다. |

### 7.4 결과는 어땠는가

요약 결과:

| 방식 | 성공률 | 평균 시간 | 평균 prompt token |
|---|---:|---:|---:|
| `symbol_bundle_oracle` | 1/3 | 2.298초 | 369 |
| `symbol_filtered_select_bundle` | 1/3 | 4.588초 | 587 |

문제별 결과:

| 문제 | bundle에 필요한 코드가 들어갔는가 | 결과 |
|---|---|---|
| expired token 에러 문자열 | 들어갔다. `validate_token`과 `TOKEN_EXPIRED_MESSAGE`가 함께 들어갔다. | 성공 |
| cache key 계산 | 들어갔다. `build_cache_key`와 `sanitize_report_name`이 함께 들어갔다. | 실패 |
| billing 상태와 수수료 | 들어갔다. `compute_late_fee`와 `invoice_status`가 함께 들어갔다. | 실패 |

왜 실패했을까?

중요한 점은 "필요한 코드를 못 찾아서" 실패한 것이 아니라는 점이다.

두 번째 문제에서는 LLM이 문자열 변환을 잘못 계산했다.

세 번째 문제에서는 LLM이 숫자 계산을 잘못했다.

즉 이렇게 볼 수 있다.

```text
필요한 종이는 잘 찾아서 책상 위에 올렸다.
그런데 LLM이 종이에 적힌 계산을 틀렸다.
```

### 7.5 이전 실험보다 좋아진 점

이전 hard test와 비교하면 크게 좋아졌다.

| 방식 | 성공률 | 평균 시간 | 평균 prompt token |
|---|---:|---:|---:|
| 기존 `symbol_oracle` | 0/3 | 2.451초 | 225 |
| 기존 `symbol_select` | 0/3 | 172.170초 | 33,922 |
| `symbol_bundle_oracle` | 1/3 | 2.298초 | 369 |
| `symbol_filtered_select_bundle` | 1/3 | 4.588초 | 587 |

특히 `symbol_filtered_select_bundle`은 기존 `symbol_select`보다 훨씬 작고 빨라졌다.

- prompt token 약 98.3% 감소
- wall time 약 97.3% 감소
- 후보 symbol 638개에서 문제별 2개에서 6개로 감소
- selector는 3문제 모두 올바른 symbol을 골랐다.

### 7.6 아주 쉽게 해석하면

Symbol bundle 실험은 필요한 코드 찾기를 잘했다.

하지만 LLM이 계산을 틀릴 수 있다는 것을 보여줬다.

그래서 코드 어시스턴트는 두 가지를 나눠야 한다.

```text
1. 필요한 코드를 찾기
2. 찾은 코드를 정확히 계산하고 해석하기
```

첫 번째는 symbol bundle이 도와줄 수 있다.

두 번째는 LLM만 믿지 말고, 실제 실행 도구나 작은 계산 도구가 도와주는 것이 좋다.

### 7.7 한계는 무엇인가

- 아직 같은 파일 안의 helper 중심으로만 잘 확장한다.
- 실제 프로젝트에서는 다른 파일의 함수나 class도 따라가야 한다.
- 질문에 파일명이나 함수명이 없으면 후보를 놓칠 수 있다.
- 실험용 Python 코드라서 실제 프로젝트보다 단순하다.
- 실제 프로젝트에는 import, decorator, inheritance 같은 복잡한 구조가 있다.

### 7.8 코드 어시스턴트에는 어떻게 쓰는가

좋은 흐름은 다음과 같다.

```text
사용자 질문을 받는다.
관련 파일 후보를 찾는다.
그 파일 안의 symbol 목록을 작게 만든다.
LLM 또는 규칙으로 대표 symbol을 고른다.
그 symbol 주변의 상수, helper, callee를 함께 묶는다.
LLM에게 작은 bundle만 준다.
계산형 문제는 실제 실행 도구로 확인한다.
답변하거나 코드를 수정한다.
```

## 8. 네 실험을 합치면 어떤 코드 어시스턴트가 되는가

네 실험을 합치면 목표는 분명하다.

```text
LLM에게 모든 것을 다 읽히는 코드 어시스턴트가 아니라,
필요한 증거만 찾아서 보여주는 코드 어시스턴트가 되어야 한다.
```

전체 흐름은 이렇게 만들 수 있다.

```text
사용자 요청
-> 외부 파일이 필요하면 source pointer 사용
-> 실행 결과가 크면 runtime pointer 사용
-> 코드가 필요하면 symbol index와 symbol bundle 사용
-> 작은 증거 묶음만 LLM에게 전달
-> 계산이 필요하면 실제 도구로 확인
-> 답변 또는 코드 수정
-> 테스트를 다시 실행하고 결과를 runtime pointer로 저장
-> 필요한 부분만 다시 꺼내서 확인
```

예를 들어 "테스트가 깨졌으니 고쳐줘"라는 요청은 이렇게 처리할 수 있다.

```text
테스트 실행
-> 긴 실패 로그 생성
-> runtime://... 번호표로 저장
-> 실패 줄만 꺼냄
-> 실패한 테스트 이름 확인
-> symbol index로 관련 함수 찾기
-> 함수와 helper를 symbol bundle로 묶기
-> LLM에게 작은 증거만 주기
-> 수정
-> 테스트 재실행
```

이 구조의 장점은 다음과 같다.

- LLM 책상이 넘칠 가능성이 줄어든다.
- 읽는 양이 줄어 시간이 줄어든다.
- 관련 없는 정보 때문에 헷갈릴 가능성이 줄어든다.
- 큰 로그와 큰 파일을 반복해서 통째로 넣지 않아도 된다.
- 코드 수정 근거가 어디서 왔는지 추적하기 쉬워진다.

## 9. 아주 쉬운 용어 설명

| 용어 | 아주 쉬운 설명 |
|---|---|
| LLM | 질문에 답하는 똑똑한 글쓰기 모델이다. |
| prompt | LLM에게 보여주는 질문이나 자료이다. |
| context window | LLM의 책상 크기이다. 한 번에 올릴 수 있는 글의 양이다. |
| token | LLM이 글을 잘게 나눠 읽는 조각이다. |
| prompt token | prompt가 몇 조각으로 나뉘었는지 센 값이다. |
| context overflow | LLM 책상보다 글이 너무 많아서 못 읽는 상황이다. |
| pointer | 큰 자료에 붙이는 번호표이다. |
| source pointer | 이미 있는 파일에 붙인 번호표이다. |
| runtime pointer | 실행 중 새로 생긴 결과물에 붙인 번호표이다. |
| code pointer | 코드 안의 함수나 클래스에 붙인 번호표이다. |
| RuntimeStore | 실행 중 생긴 큰 결과물을 잠시 넣어 두는 보관함이다. |
| extraction tool | 큰 자료에서 필요한 부분만 꺼내는 도구이다. |
| selector | 여러 후보 중 무엇을 볼지 고르는 역할이다. |
| oracle | 정답 위치를 이미 안다고 치고 하는 실험 조건이다. |
| symbol | 코드 안의 이름 붙은 조각이다. 함수, 클래스, 상수 등이 있다. |
| symbol index | 코드 조각들이 어디 있는지 적어 둔 목록이다. |
| symbol bundle | 필요한 코드 조각과 주변 helper를 함께 묶은 작은 꾸러미이다. |
| Self-Extend | LLM의 책상을 더 크게 만들어 긴 글을 받게 하는 방법이다. |
| group attention | Self-Extend를 켤 때 쓴 설정이다. 실험에서는 G=2, G=4로 불렀다. |
| PPL | 모델이 글을 얼마나 자연스럽게 예측하는지 보는 점수이다. 낮을수록 좋다. |
| PG-19 | 긴 문학 글로 만든 평가 자료이다. |
| quantization | 모델을 더 작고 빠르게 만들기 위해 숫자를 더 간단하게 저장하는 방법이다. |
